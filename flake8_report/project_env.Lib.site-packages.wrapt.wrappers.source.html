670 def __init__(self, wrapped, wrapper, enabled=None):
671 # What it is we are wrapping here could be anything. We need to
672 # try and detect specific cases though. In particular, we need
673 # to detect when we are given something that is a method of a
674 # class. Further, we need to know when it is likely an instance
675 # method, as opposed to a class or static method. This can
676 # become problematic though as there isn't strictly a fool proof
677 # method of knowing.
678 #
679 # The situations we could encounter when wrapping a method are:
680 #
681 # 1. The wrapper is being applied as part of a decorator which
682 # is a part of the class definition. In this case what we are
683 # given is the raw unbound function, classmethod or staticmethod
684 # wrapper objects.
685 #
686 # The problem here is that we will not know we are being applied
687 # in the context of the class being set up. This becomes
688 # important later for the case of an instance method, because in
689 # that case we just see it as a raw function and can't
690 # distinguish it from wrapping a normal function outside of
691 # a class context.
692 #
693 # 2. The wrapper is being applied when performing monkey
694 # patching of the class type afterwards and the method to be
695 # wrapped was retrieved direct from the __dict__ of the class
696 # type. This is effectively the same as (1) above.
697 #
698 # 3. The wrapper is being applied when performing monkey
699 # patching of the class type afterwards and the method to be
700 # wrapped was retrieved from the class type. In this case
701 # binding will have been performed where the instance against
702 # which the method is bound will be None at that point.
703 #
704 # This case is a problem because we can no longer tell if the
705 # method was a static method, plus if using Python3, we cannot
706 # tell if it was an instance method as the concept of an
707 # unnbound method no longer exists.
708 #
709 # 4. The wrapper is being applied when performing monkey
710 # patching of an instance of a class. In this case binding will
711 # have been perfomed where the instance was not None.
712 #
713 # This case is a problem because we can no longer tell if the
714 # method was a static method.
715 #
716 # Overall, the best we can do is look at the original type of the
717 # object which was wrapped prior to any binding being done and
718 # see if it is an instance of classmethod or staticmethod. In
719 # the case where other decorators are between us and them, if
720 # they do not propagate the __class__ attribute so that the
721 # isinstance() checks works, then likely this will do the wrong
722 # thing where classmethod and staticmethod are used.
723 #
724 # Since it is likely to be very rare that anyone even puts
725 # decorators around classmethod and staticmethod, likelihood of
726 # that being an issue is very small, so we accept it and suggest
727 # that those other decorators be fixed. It is also only an issue
728 # if a decorator wants to actually do things with the arguments.
729 #
730 # As to not being able to identify static methods properly, we
731 # just hope that that isn't something people are going to want
732 # to wrap, or if they do suggest they do it the correct way by
733 # ensuring that it is decorated in the class definition itself,
734 # or patch it in the __dict__ of the class type.
735 #
736 # So to get the best outcome we can, whenever we aren't sure what
737 # it is, we label it as a 'function'. If it was already bound and
738 # that is rebound later, we assume that it will be an instance
739 # method and try an cope with the possibility that the 'self'
