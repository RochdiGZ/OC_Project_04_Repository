1963

2028 """
2029 Context manager and decorator doing much the same as
2030 ``warnings.catch_warnings``.
2031  
2032 However, it also provides a filter mechanism to work around
2033 https://bugs.python.org/issue4180.
2034  
2035 This bug causes Python before 3.4 to not reliably show warnings again
2036 after they have been ignored once (even within catch_warnings). It
2037 means that no "ignore" filter can be used easily, since following
2038 tests might need to see the warning. Additionally it allows easier
2039 specificity for testing warnings and can be nested.
2040  
2041 Parameters
2042 ----------
2043 forwarding_rule : str, optional
2044 One of "always", "once", "module", or "location". Analogous to
2045 the usual warnings module filter mode, it is useful to reduce
2046 noise mostly on the outmost level. Unsuppressed and unrecorded
2047 warnings will be forwarded based on this rule. Defaults to "always".
2048 "location" is equivalent to the warnings "default", match by exact
2049 location the warning warning originated from.
2050  
2051 Notes
2052 -----
2053 Filters added inside the context manager will be discarded again
2054 when leaving it. Upon entering all filters defined outside a
2055 context will be applied automatically.
2056  
2057 When a recording filter is added, matching warnings are stored in the
2058 ``log`` attribute as well as in the list returned by ``record``.
2059  
2060 If filters are added and the ``module`` keyword is given, the
2061 warning registry of this module will additionally be cleared when
2062 applying it, entering the context, or exiting it. This could cause
2063 warnings to appear a second time after leaving the context if they
2064 were configured to be printed once (default) and were already
2065 printed before the context was entered.
2066  
2067 Nesting this context manager will work as expected when the
2068 forwarding rule is "always" (default). Unfiltered and unrecorded
2069 warnings will be passed out and be matched by the outer level.
2070 On the outmost level they will be printed (or caught by another
2071 warnings context). The forwarding rule argument can modify this
2072 behaviour.
